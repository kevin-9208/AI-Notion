写 AI 提示词（Prompt），本质上不是“写一句话给 AI”，而是在做一种**自然语言编程（Natural Language Programming）**。

你可以把模型理解成一个：

* **知识压缩器**（知道很多）
* **概率生成器**（猜你想要什么）
* **上下文解释器**（非常依赖你给的信息）

所以提示词质量，直接决定输出质量。

---

# 一、先理解：为什么提示词重要？

同样一个需求：

用户A：

> 写一个登录页面

用户B：

> 用 Vue3 + TailwindCSS 写一个响应式登录页，包含邮箱、密码、记住我、Google OAuth 按钮，深色模式支持，组件化拆分，适合 Supabase Auth 集成。

结果差距会非常大。

因为 AI 最怕：

* 模糊目标
* 缺上下文
* 缺约束
* 缺标准

提示词的作用就是：

把你的脑内模糊意图，压缩成 AI 可执行指令。

这就是核心。

---

# 二、提示词的底层结构（万能公式）

一个高质量 Prompt：

> 任务 + 上下文 + 约束 + 输出格式 + 标准

公式：

```text
[Role]
你是谁

[Task]
你要做什么

[Context]
背景信息是什么

[Constraints]
限制条件是什么

[Output Format]
输出格式是什么

[Quality Standard]
做到什么程度算合格
```

例子：

```text
你是一个资深 Vue3 工程师。

帮我写一个旅游规划系统的 Trips CRUD 模块。

技术栈：
- Vue3
- Pinia
- Supabase
- TailwindCSS

要求：
- Composition API
- TypeScript
- 可维护
- 模块化
- 错误处理完整

输出：
1. store
2. composable
3. page
4. component

代码必须是生产级。
```

这个结构非常稳定。

# 三、Prompt 的六大核心维度

---

# 1. Role（角色设定）

模型会根据角色切换行为模式。

比如：

开发：

```text
你是一个高级全栈工程师
```

产品：

```text
你是一个资深产品经理
```

设计：

```text
你是一个UI/UX设计专家
```

写作：

```text
你是一个专业科幻小说作家
```

心理：

```text
你是荣格心理学研究者
```

作用：

角色 = 决定视角

比如：

“解释数据库索引”

普通：

> 解释数据库索引

专家：

> 你是数据库内核工程师，深入解释 B+ Tree 索引实现原理

深度完全不同。

---

# 2. Task（任务清晰）

坏：

```text
帮我优化代码
```

好：

```text
帮我找出这个 Vue3 组件中的性能瓶颈，并优化渲染逻辑。
```

任务要：

* 明确动作
* 明确对象
* 明确目标

公式：

```text
动词 + 对象 + 目标
```

例：

* 分析这个项目架构
* 重构这段代码
* 优化 SQL 查询
* 设计数据库 schema
* 写测试用例

---

# 3. Context（上下文）

这个是高手和普通用户最大差距。

AI 非常吃上下文。

比如：

坏：

```text
写个后台
```

好：

```text
做一个旅游规划 SaaS 后台：

用户：
自由行游客

功能：
- 行程管理
- 地图标记
- 预算管理
- 协作分享

技术：
Vue3 + Supabase

目前已完成：
Auth、Trips 模块
现在做 Budget 模块
```

这会极大提高连续性。

---

# 4. Constraints（约束）

没有约束，AI 会自由发挥。

有时候自由发挥 = 偏题。

例：

```text
限制：
- 不要使用第三方状态库
- 不要用 Options API
- 必须 TypeScript
- 必须 SSR 兼容
- 优先考虑性能
```

常见约束：

技术约束：

* 用 React/Vue/Svelte
* 用 Python/Go/Rust

风格约束：

* 极简
* 工业风
* 苹果风

性能约束：

* O(n)
* 避免重复渲染

业务约束：

* 面向新手用户
* 支持多人协作

---

# 5. Output Format（输出格式）

这是被严重低估的部分。

比如：

坏：

```text
给我方案
```

好：

```text
按以下格式输出：

1. 问题分析
2. 架构设计
3. 数据库设计
4. API设计
5. 前端实现
6. 风险点
```

代码：

```text
输出完整目录结构
```

学习：

```text
用表格输出
```

比较：

```text
按优缺点对比
```

这会让结果稳定很多。

---

# 6. Quality Standard（质量标准）

非常强。

比如：

```text
不要给 demo 级代码，要给生产级代码。
```

或者：

```text
从架构层深入，不要停留在表面。
```

或者：

```text
假设读者是中级开发者。
```

或者：

```text
像源码作者一样解释。
```

质量标准决定深度。

# 四、高阶技巧（高手区）

---

# 技巧1：Few-shot（示例驱动）

你给例子，AI 学你的风格。

比如：

```text
按照这个风格继续：

输入：
用户登录失败

输出：
原因分析：
1...
2...

解决方案：
1...
2...
```

这叫示例学习。

很强。

---

# 技巧2：Chain of Thought（链式思考）

让 AI 分步骤思考。

坏：

```text
帮我设计系统
```

好：

```text
按步骤分析：

1. 明确需求
2. 找出核心实体
3. 设计数据库
4. 设计 API
5. 设计前端状态流
```

复杂任务提升巨大。

---

# 技巧3：Self-critique（自检）

非常强。

```text
完成后请检查：

- 是否有性能问题
- 是否有安全问题
- 是否有边界条件遗漏
```

代码质量会明显提高。

---

# 技巧4：Iterative Prompting（迭代提示）

高手基本不用“一发入魂”。

流程：

第一轮：

```text
先给整体架构
```

第二轮：

```text
展开数据库设计
```

第三轮：

```text
展开 Auth 模块
```

第四轮：

```text
展开错误处理
```

这才是正确玩法。

可以。上面讲的是“结构”，现在讲更偏实战的 **通用 Prompt 技巧（Prompt Skills）**。

这些技巧基本跨模型通用（OpenAI 的 GPT、Anthropic 的 Claude、Google 的 Gemini 都适用），只是效果权重略有差异。

---

# 一、信息控制类（最重要）

这是最核心的一组。

---

# 1. 信息增量原则（Incremental Context）

不要一次把所有东西砸进去。

坏：

```text
帮我做一个完整 SaaS，从数据库到前端全部生成。
```

问题：

* 容易漂移
* 容易偷懒
* 容易失去一致性

好：

```text
第一步，只做数据库设计。
```

然后：

```text
基于这个数据库设计 API。
```

然后：

```text
基于 API 设计前端 store。
```

原则：

> 大任务拆成状态机。

这个非常重要。

---

# 2. 上下文锚定（Context Anchoring）

避免 AI 忘记重点。

例：

```text
记住以下原则，后续所有回答都必须遵守：

1. 技术栈固定 Vue3 + Supabase
2. 优先简单可维护
3. 不引入重依赖
4. 默认移动端优先
```

作用：

建立长期上下文约束。

像给模型加系统配置。

---

# 3. 显式优先级（Priority Ordering）

模型不知道什么最重要。

你要告诉它：

```text
优先级：

P0：先保证正确性
P1：再保证可维护性
P2：最后优化性能
```

或者：

```text
优先级：

1. 安全性最高
2. 用户体验第二
3. 开发效率第三
```

这会明显改变输出方向。

---

# 二、深度控制类

决定回答“浅”还是“深”。

---

# 4. 深度拉杆（Depth Lever）

一句话就能改变层次：

浅：

```text
简单解释
```

中：

```text
深入解释
```

深：

```text
从底层原理解释
```

更深：

```text
像实现者一样解释
```

最深：

```text
像框架作者一样解释设计动机、权衡和 tradeoff
```

这个非常强。

比如：

解释 Supabase：

普通：

> 是个开源 Firebase 替代品

深度模式：

> 从 Postgres WAL、Realtime replication、RLS 机制展开

完全不是一个层级。

---

# 5. 多层展开（Progressive Expansion）

这个特别适合学习。

```text
先用一句话解释。
再用一段话解释。
最后从底层展开。
```

例：

学习 Photosynthesis：

第一层：

> 植物把光变成能量。

第二层：

> 利用叶绿体把 CO₂ 和水转成葡萄糖。

第三层：

> 展开电子传递链、ATP/NADPH。

理解速度会快很多。

---

# 三、纠错增强类

---

# 6. 对抗式验证（Adversarial Review）

让 AI 自己攻击自己：

```text
给出方案后，请站在 reviewer 角度批判它。
```

或者：

```text
找出这个方案未来可能崩的地方。
```

特别适合架构设计。

---

# 7. 反例驱动（Counterexample Driven）

非常强。

```text
举三个失败案例说明这个方案为什么不好。
```

例：

```text
为什么不要把所有状态都放在 Pinia？
请举反例。
```

能快速理解边界。

---

# 8. 假设攻击（Stress Test）

问：

```text
如果用户量扩大100倍会怎样？
```

或者：

```text
如果数据库延迟突然增加10倍？
```

这是架构师思维。

---

# 四、认知增强类

提升学习效率。

---

# 9. 类比映射（Analogy Mapping）

神技。

```text
用现实世界类比解释。
```

例：

解释 Neural Network：

> 像一个层层筛选信息的工厂。

解释数据库事务：

> 像银行转账流程。

抽象概念秒变具体。

---

# 10. 第一性原理拆解（First Principles）

非常强。

```text
从第一性原理拆解这个问题。
```

例：

为什么缓存有用？

不是：

> 因为快。

而是：

* CPU 和内存速度差
* 内存和磁盘速度差
* 数据局部性

这就是底层理解。

---

# 11. 本质提问（Essence Extraction）

问：

```text
这个东西本质上是什么？
```

例：

Vue.js 本质是什么？

答案不是：

> 前端框架。

而是：

> 状态到 UI 的映射系统。

这种问题很值钱。

# 12. 强制结构化

```text
按表格输出
```


# 13. 信心标记（Confidence Tagging）

减少幻觉：

```text

高确定 / 中确定 / 低确定
```

非常适合：

* 法律
* 医疗
* 新技术

# 14. 区分事实和推测

这个很重要：

```text
把“事实”、“推断”、“建议”分开写。
```

能过滤很多噪音。

# 六、开发者专用神技

这个最适合你。

---

# 技巧A：Code Review 模式

```text
像 senior reviewer 一样 review。
```

比：

> 看看代码有没有问题

强太多。

---

# 技巧B：架构升级模式

```text
这个项目未来变大，哪里最先崩？
```

这是进阶最快的方式。

---

# 技巧C：Tradeoff 分析

神技。

```text
不要只给方案，分析 tradeoff。
```

例：

比较：

* Supabase
* Firebase

从：

* 成本
* 锁定风险
* 扩展性
* 性能

分析。

这是架构核心能力。

---

# 技巧D：决策模拟

```text
如果你是 Tech Lead，你会选哪个？
为什么？
```

这个能直接学决策逻辑。

非常值钱。

---

最强通用模板（万能增强器）：

```text
回答时请：

1. 先讲本质
2. 再讲原理
3. 再讲实现
4. 再讲 tradeoff
5. 再讲边界条件
6. 再讲失败案例
7. 最后讲最佳实践
```

这条加到任何问题后面，质量通常都会上一个台阶。

这个属于真正通用的 Prompt skill。不是某个平台技巧，而是通用认知杠杆。
